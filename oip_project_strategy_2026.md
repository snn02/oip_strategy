# Проектное направление ОИП

**Ответственный:** Виктория Писаревская

**Горизонт планирования:** 2026 год

Документ описывает стратегию, цели, OKR и KPI проектного направления
Отдела интернет‑проектов (ОИП).

Документ служит управленческой рамкой для руководителя проектного направления
и помогает ответить на три ключевых вопроса:

1. Зачем существует проектное направление.
2. Какие изменения должны произойти в 2026 году.
3. По каким показателям оценивается результат.

Проектное направление — **основной источник валовой прибыли ОИП**
и главный драйвер роста клиентской базы сопровождения.

Руководитель направления управляет:

- экономикой проектов
- содержанием проектов
- сроками реализации
- загрузкой производства на проекте
- рисками проектов
- передачей завершённых проектов в сопровождение

---------------------------------------------------------------------

## Связь с другими направлениями ОИП

| Направление | Роль | Интерфейс с проектным направлением |
|---|---|---|
| Продажи (Лена) | Поставляет поток проектов | Совместно формирует оценку и ценовой коридор |
| Производство (Сербин) | Выполняет проектные работы | Проектное направление управляет загрузкой и приоритетами |
| Сопровождение (Ира) | Принимает завершённые проекты | Конверсия проектов в сопровождение — KPI Виктории |
| СМБ (Дима) | Выполняет короткие задачи | Граница: задачи свыше 80 часов передаются в проектное направление |

# 1. Стратегические цели направления

Цели отвечают на вопрос **«зачем существует проектное направление»**
и задают контекст для OKR и KPI.

---------------------------------------------------------------------

## Цель 1. Проектное направление — основной источник валовой прибыли ОИП

Проекты должны генерировать основную часть прибыли ОИП
и обеспечивать выполнение финансовых целей направления.

Это достигается через:

- управляемую оценку проектов
- контроль трудозатрат
- управление содержанием проекта
- своевременное завершение проектов
- предотвращение перерасхода бюджета.

Проект считается успешным, когда:

- результат принят заказчиком в согласованные сроки
- проект принёс плановую валовую прибыль
- зафиксированы уроки для будущих проектов

---------------------------------------------------------------------

## Цель 2. Проекты управляются через прозрачное содержание работ

Главная причина провалов проектов — отсутствие управляемого содержания.
Каждый проект должен иметь измеримую модель требований,
связанную с оценкой трудозатрат.

Каждый проект включает:

- чёткую цель
- понятный образ результата
- границы работ
- критерии приёмки
- декомпозицию требований в задачи с оценкой
- контрольные точки результата
- условия использования результата проекта.

Это позволяет:

- корректно оценивать и прогнозировать трудозатраты
- управлять изменениями требований и ростом содержания
- управлять ожиданиями заказчика в рамках бюджета

---------------------------------------------------------------------

## Цель 3. Проектное направление обеспечивает стабильную загрузку производства

Проекты являются основным источником загрузки производственной команды (Саша Сербин).
Задача руководителя — поддерживать предсказуемый поток работ без простоев и перегрузок.

Это достигается через:

- прогноз загрузки производства на горизонт 2 месяцев
- своевременный запуск проектов
- координацию с продажами по работе с пресейлами

---------------------------------------------------------------------

## Цель 4. Завершённые проекты передаются в сопровождение

Проектное направление отвечает за полный жизненный цикл проекта —
от запуска до передачи клиента в сопровождение (Ира).

Это означает:

- каждый завершённый проект имеет явное решение о дальнейшей работе с клиентом
- проект закрывается актом и формально передаётся в сопровождение или завершается
- конверсия проектов в сопровождение — один из ключевых показателей роста ОИП

---------------------------------------------------------------------

## Цель 5. Проектное направление масштабируется без роста управленческого хаоса

Рост количества проектов не должен приводить к росту управленческой нагрузки
на руководителя ОИП и офиса.

Проекты ведутся по единым правилам:

- единая система задач и структура проектов
- единый классификатор типов проектов и шаблоны планов
- единая логика анализа показателей
- регулярные ретроспективы и накопление базы знаний

---------------------------------------------------------------------

# 2. OKR проектного направления

OKR фиксируют изменения системы управления проектами,
которые должны произойти в 2026 году.

OKR — это не план задач, а описание **нового состояния системы**.

---------------------------------------------------------------------

# Objective 1

## Проекты приносят плановую валовую прибыль

Проектное направление управляет не только задачами, но и экономикой проектов.
Каждый проект должен закрываться с прибылью,
а проблемы экономики — выявляться до завершения проекта.

### KR1 — Выполнение финансового плана

План по валовой прибыли проектов выполняется **≥ 100%**.

Метрика отражает итоговый результат работы всего портфеля проектов.

---------------------------------------------------------------------

### KR2 — Прибыльность проектов

Не менее **90% завершённых проектов** имеют положительную маржу (в терминах планов проектов ЕИС).

Защищает от ситуации, когда несколько крупных проектов маскируют убытки по другим.

---------------------------------------------------------------------

### KR3 — Точность оценки проектов

Отклонение фактических трудозатрат от плановой оценки
не превышает **±20% по портфелю**.

Показывает зрелость системы управления содержанием и качество оценки трудозатрат.

---------------------------------------------------------------------

### KR4 — Соблюдение сроков проектов

Не менее **90% проектов** завершаются в согласованный срок ±10%.

Отражает управляемость сроков и способность команды контролировать ход проекта.

---------------------------------------------------------------------

# Objective 2

## Проектное направление управляет содержанием, бюджетом, сроками и ожиданиями клиента

Руководитель направления должен в любой момент времени понимать:

- что именно должно быть сделано в проекте
- сколько работы уже выполнено и сколько осталось
- укладывается ли проект в бюджет
- когда проект будет завершён
- какие риски могут повлиять на сроки и бюджет

Это позволяет управлять ожиданиями заказчика и экономикой проектов.

### KR1 — Формализованное содержание проекта

**100% проектов имеют формализованное содержание проекта.**

Содержание проекта включает:

- цели проекта
- образ результата
- бюджет в трудозатратах и деньгах
- границы работ
- критерии приёмки результата
- показатели использования результата
- ключевые контрольные точки проекта

Наличие формализованного содержания позволяет:

- синхронизировать ожидания заказчика и команды
- корректно оценивать трудозатраты
- управлять изменениями требований
- прогнозировать сроки и экономику проекта

---------------------------------------------------------------------

### KR2 — Прозрачность требований проекта

**По всем проектам поддерживается актуальная картина требований.**

Для каждого проекта в любой момент времени известно:

- сколько требований всего
- сколько требований оценено
- сколько требований выполнено
- сколько требований поступило
- сколько требований находится в работе

Это позволяет:

- управлять ростом содержания проекта (scope creep)
- понимать реальный объём оставшихся работ
- корректно прогнозировать сроки и бюджет проекта

---------------------------------------------------------------------

### KR3 — Управляемость бюджета проекта

**По всем проектам существует актуальный прогноз итоговых трудозатрат до завершения (EAC).**

Для каждого проекта в любой момент времени известны:

- фактические трудозатраты
- оценка оставшихся работ
- прогноз итоговой трудоёмкости проекта

Это позволяет:

- своевременно выявлять риск перерасхода бюджета
- принимать управленческие решения по корректировке содержания
- обсуждать изменения объёма работ с заказчиком

---------------------------------------------------------------------

### KR4 — Управляемость сроков проекта

**Отклонение прогнозируемой даты завершения от фактической не превышает ±20% по портфелю.**

Метрика отражает способность команды:

- своевременно выявлять отклонения проекта от плана
- корректно прогнозировать завершение проекта
- управлять ожиданиями заказчика

Высокая точность прогноза означает, что команда понимает
реальный объём работ и контролирует ход проекта.

---------------------------------------------------------------------

### KR5 — Управление рисками проекта

**По всем проектам ведётся реестр ключевых рисков,
и критические риски выявляются до того, как приводят к срыву сроков или бюджета.**

Для каждого проекта:

- определены основные риски проекта
- назначены ответственные за управление рисками
- определены действия по управлению риском

Это позволяет выявлять проблемы проекта на ранней стадии
и предотвращать срывы сроков и перерасход бюджета.

---------------------------------------------------------------------

# Objective 3

## Проектное направление обеспечивает предсказуемую загрузку производства

Задача — сделать загрузку команды Саши Сербина управляемой и прогнозируемой.
Проектное направление является главным источником загрузки производства.

### KR1 — Уровень загрузки команды

Средняя загрузка производственных специалистов на проектных работах
составляет **70–85%**.

- Ниже 70% — сигнал проблемы с потоком проектов
- Выше 85% — риск срыва сроков и выгорания команды

---------------------------------------------------------------------

### KR2 — Прогноз загрузки производства

По портфелю проектов существует прогноз загрузки **до завершения каждого проекта**.

Прогноз обновляется до каждого проектного комитета.

Это позволяет управлять распределением ресурсов.

---------------------------------------------------------------------

### KR3 — Снижение необъяснённых простоев

Доля простоев специалистов, выделелнных на проекты, 
без зафиксированной причины
составляет **менее 5% рабочего времени**.

---------------------------------------------------------------------

# Objective 4

## Проектная система масштабируется и передаёт клиентов в сопровождение

Рост портфеля не должен снижать управляемость.
Каждый завершённый проект — потенциальный клиент сопровождения.

### KR1 — Управляемая ёмкость команды

Проектная команда может одновременно вести:

- 1 крупный проект
- 2 средних проекта

Добавление проектов сверх ёмкости требует согласования с руководителем ОИП.

Это фиксирует реальную пропускную способность и предотвращает перегрузку.

---------------------------------------------------------------------

### KR2 — Быстрый запуск проектов

Срок запуска проекта после подписания договора не превышает **5 рабочих дней**.

Быстрый старт снижает риск снижения вовлеченности заказчика и команды, 
уменьшает межпроектные простои.

---------------------------------------------------------------------

### KR3 — Стандартизация проектов

Разработан классификатор типов проектов, определнные методологией ппродаж ОИП.

Для каждого типа созданы типовые шаблоны проектных планов и оценок для каждой глубины пресейла.

Это повышает скорость оценки, снижает трудозатраты на пресейл, повышает конверсию пресейла в проект.

---------------------------------------------------------------------

### KR4 — Передача проектов в сопровождение

**По 100% завершённых проектов принято и зафиксировано решение:**
передан в сопровождение / завершён без сопровождения.

Если проект закрывается без сопровождения, 
то фиксируется причина, которая анализируется в урока проекта.

Руководитель проектного направления инициирует передачу проекта команде сопровождения.

Это закрывает жизненный цикл проекта
и обеспечивает рост клиентской базы сопровождения.

---------------------------------------------------------------------

### KR5 — Обучение на опыте проектов

По каждому завершённому проекту проводится ретроспектива.

Зафиксированные уроки используются для улучшения оценки
и управления следующими проектами.

---------------------------------------------------------------------

# 3. KPI руководителя проектного направления

KPI оцениваются **ежеквартально** и влияют на расчёт премии.

> Загрузка производства не является KPI — это диагностическая метрика,
> которая смотрится на проектном комитете раз в 2 недели.

| KPI | Показатель | Формула / Метрика | Цель |
|---|---|---|---|
| KPI 1 | Валовая прибыль проектов | Выручка проектов – себестоимость | ≥ 80% плана (порог премии) |
| KPI 2 | Точность оценки проектов | Факт часы / оценка по портфелю | Отклонение ≤ ±20% |
| KPI 3 | Доля проектов, завершённых в срок | Проекты в срок / все завершённые | ≥ 80% |
| KPI 4 | Передача проектов в сопровождение | Проекты с решением о судьбе / завершённые | 100% проектов |

---------------------------------------------------------------------

## KPI 1 — Валовая прибыль проектов

Ключевой показатель эффективности проектного направления.
Отражает итоговую экономику портфеля проектов.

Формула:

**ВП = Выручка проектов – прямая себестоимость**

Порог включения премии:

**ВП ≥ 80% от плана**

---------------------------------------------------------------------

## KPI 2 — Точность оценки проектов

Показывает качество управления содержанием проектов
и зрелость системы оценки трудозатрат.

Формула:

**фактические трудозатраты / плановая оценка по портфелю**

Норма: отклонение **≤ ±20%**

Если показатель хуже нормы — анализируется управление требованиями и scope creep.

---------------------------------------------------------------------

## KPI 3 — Доля проектов, завершённых в срок

Отражает управляемость сроков и способность команды контролировать ход проектов.

Метрика:

**проекты завершённые в срок / все завершённые проекты**

Норма: **≥ 80%**

---------------------------------------------------------------------

## KPI 4 — Передача проектов в сопровождение

Фиксирует завершённость жизненного цикла проекта
и вклад в рост направления сопровождения.

Метрика:

**по 100% завершённых проектов принято явное решение о дальнейшей работе с клиентом**

Варианты решения:

- передан в сопровождение к Ире
- завершён без сопровождения (зафиксирована причина)

---------------------------------------------------------------------

# 4. Управленческая логика направления

## Зона ответственности Виктории

Руководитель проектного направления отвечает за:

- измеримость содержания проектов (требования → задачи → часы)
- управление реализацией проектов и прогнозирование бюджета
- экономику портфеля проектов
- прогноз загрузки и координацию с производством (Сербин)
- передачу завершённых проектов в сопровождение (Ира)

---------------------------------------------------------------------

## Матрица ответственности за экономику проекта

Экономика проекта формируется в двух зонах ответственности:
**до подписания договора** (Лена) и **после** (Виктория).

| Элемент | Продажи (Лена) | РП (Виктория) | Производство (Сербин) |
|---|---|---|---|
| Выбор клиента и проекта | Отвечает | Участвует | Информируется |
| Формирование цены проекта | Отвечает | Утверждает min-цену | — |
| Оценка трудозатрат | Участвует | Отвечает | Участвует |
| Управление бюджетом и сроками | Информируется | Отвечает | Участвует |
| Качество и скорость разработки | — | Контролирует | Отвечает |
| Передача в сопровождение | Участвует | Отвечает | — |

> Оценку трудозатрат вправе менять только РП.
> Цену — только продажи.
> Продажи не могут продавать ниже минимальной цены, рассчитанной РП.

---------------------------------------------------------------------

## Цепочка управления проектом

Управление проектным направлением строится по цепочке причин и следствий:

```
Требования (Scope)
        ↓
Оценка (Effort)
        ↓
Прогноз сроков (Schedule)
        ↓
Контроль бюджета (Cost)
        ↓
Валовая прибыль (Profit)
```

**Если не управлять содержанием — неизбежно ломаются оценки, сроки и экономика.**

Диагностика при падении валовой прибыли:

- ВП падает → смотрим точность оценки
- Оценки плывут → смотрим управление требованиями (Scope Stability, Coverage)
- Требования нестабильны → смотрим дисциплину фиксации изменений (Change Request)
- Сроки срываются → смотрим прогноз трудозатрат (EAC) и SPI/CPI на комитете

---------------------------------------------------------------------

# 5. Проектный комитет

Проектный комитет проводится **раз в 2 недели**.

Формат — быстрый обзор состояния портфеля по 8 показателям здоровья проектов.

| Блок | Показатель | Норма / Интерпретация |
|---|---|---|
| Портфель | Валовая прибыль портфеля | ≥ 80% плана |
| Портфель | Загрузка производства | 70–85% |
| Портфель | Прогноз загрузки (до завершения каждого проекта) | Наличие прогноза по всем проектам |
| Проект | Coverage (оценённые задачи / все) | ≥ 80% норма; < 70% риск |
| Проект | SPI (Schedule Performance Index) | ≥ 0.9 норма; < 0.9 отставание |
| Проект | CPI (Cost Performance Index) | ≥ 0.9 норма; < 0.9 перерасход |
| Проект | Прогноз трудозатрат (EAC) | EAC ≤ бюджет проекта |
| Проект | Прогноз даты завершения | Отклонение ≤ ±20% от плана |

**Цветовая модель здоровья проекта:**

- 🟢 **Зелёный** — все показатели в норме; проект управляем
- 🟡 **Жёлтый** — один показатель вне нормы; требует внимания
- 🔴 **Красный** — два и более показателя вне нормы; нужны управленческие действия

> Детальная методология расчёта метрик здоровья проектов описана
> в отдельном документе `oip_project_metrics_2026.md`
